## 1. Hypothesis verdicts

### H1 — "spamprobe weighted the sender EMAIL ADDRESS more; spamlite dilutes it by splitting From into overlapping word-tokens" → **PARTIAL (effect real, stated mechanism REFUTED)**

The observed effect ("one ham retrain flips all future mail from that sender") is real, but the user's proposed mechanism is wrong in two ways:

- **spamlite does NOT split From into colliding word-tokens.** `push_addr` emits exactly the whole address (with `@`) and the whole domain, both under the isolated `h:from:` prefix that never collides with body tokens — `tokenizer.rs:182-191`. It is *spamprobe* that splits more (`addTokenParts`, `TraditionalMailMessageParser.cc:115-143`).
- **The true sender difference is a MISSING generalizing anchor, not dilution.** Single strongest evidence: spamprobe's `addTokenParts` (`TraditionalMailMessageParser.cc:115-143`) explodes `partners.intrepidtravel.com` into the bare second-level label `intrepidtravel` — a durable token shared across every ESP subdomain/envelope the supplier ever uses. spamlite emits only the literal full subdomain (which rotates every send) plus the exact domain, so its sender memory never generalizes.
- **The "flip" itself is delivered by TRAINING-COUNT semantics, not From weighting.** spamprobe's `good`/`spam` commands run `classifyMessage`, which re-adds the retrained message's tokens up to 25× until it scores confidently ham (`SpamFilter.cc:460-491`), each ham count doubled by `goodBias=2` (`SpamFilter.cc:117`). Neither system gives the From field special combine-time weight.

### H2 — "spamprobe is more conservative from an empty/light corpus" → **CONFIRMED**

Strongest single evidence: spamprobe pins any token with fewer than `minWordCount=5` observations to a **ham-leaning `newWordScore=0.3`** (`SpamFilter.cc:122-127`, `FilterConfig.cc:39`). spamlite ships the equivalent gate **disabled** (`min_word_count=0`, `classifier.rs:220` + default at `:51`), so a token seen once in spam yields `raw_p=1.0` → `fw≈0.82` from a single sighting (`classifier.rs:217-231`). The Fisher combiner then *amplifies* those overconfident singletons: verified, a uniform field of weak `0.70` tokens scores `0.70` under spamprobe's geometric mean at any token count but `0.912 / 0.960 / 0.996` under spamlite's Fisher at 20/50/150 tokens (`classifier.rs:253-268` vs `SpamFilter.cc:367-387`).

**Cross-cutting root cause the reviews converge on:** the un-retrainable `1.000000` pin is primarily a **combiner** problem (Fisher saturates and leaves no margin a threshold can rescue), compounded by **single-shot symmetric +1 training** with every ham-protecting knob spamlite already owns turned OFF. The shared ESP tokens are *well-observed* (n≥5) in the spam corpus, so pure cold-start/thin-token fixes help H2 but do little for THIS FP — the FP needs the combiner + training-leverage + a generalizing anchor.

## 3. Recommended sequencing

**Quick wins (low effort, ship behind the harness/sweep):**
1. **#13 presence regression tests** — free, protective, do immediately.
2. **#3 `good_bias` sweep toward 1.5–2.0** — one value, already plumbed; sweep per-user via the existing nightly tuner, re-check threshold.
3. **#12 length guarantee** — fold into #2.

**Structural fixes (medium effort, the actual FP cure — each MUST be A/B'd on admin_maildir first):**
4. **#1 geometric combiner + #6 min-distance floor** — the single highest-leverage change; A/B vs Fisher on admin_maildir **and the supplier's real mailbox** before flipping the default. Do not blind-flip.
5. **#2 + #5 sender anchors + per-sender headers (+#12)** — ship together; they are synergistic (#2 handles supplier-domain-in-From, #5 handles ESP-rewritten From).
6. **#4 convergence loop** — only after the **message-digest idempotency guard** lands; FN-check on admin_maildir.

**Cold-start polish (config sweeps; pick the mutually-exclusive pairs):**
7. **#7 new_word_score split**, then **#8 gate OR #9 adaptive strength** (not both), plus **#10**.
8. **#11 tuner cold-start objective** — build the train-subset count recomputation first; this is what stops the tuner from re-creating the problem nightly.

**MUST be validated on admin_maildir via the spamlite-eval harness before shipping:** #1 (FP/FN A/B vs Fisher), #3 (FP/FN trade), #4 (FN + poisoning guard), #7/#8/#9 (FN on thin per-user corpora), #11. The combiner change in particular cannot be judged on aggregate error count alone — one catastrophic permanent FP on a high-value sender outweighs many scattered borderline misses, so weight FP cost explicitly in the eval.

## 4. Do NOT cargo-cult

- **"Fisher is strictly better than Robinson" is unproven and false for this FP.** The plan doc (`_doc/2026-04-14-spamlite-improvement-plan.md:15,227`) *asserts* this with no logged geometric-vs-Fisher benchmark; the `50>27 / 105 vs 119 errors` figure (`:150-156`) is about the term **cap**, not the combiner. Fisher is the *only* one of the two that can pin a many-weak-token message at 1.0 and make it unretrainable. (But don't blindly flip to geometric either — A/B it.)
- **Do NOT cut the term cap to 27.** Verified counterproductive: spamlite truncates by `|fw-0.5|` descending, so a smaller cap drops the weak **ham** tokens first — a real `truncate(27)` on the supplier case went `0.947→0.984` (worse). spamprobe's 27-cap only works because its tight `newWordScore`/`minWordCount` selection keeps the average on extremes.
- **Do NOT port spamprobe's frequency training.** spamlite already does the better thing — presence/Bernoulli at both train and score (`tokenizer.rs:170-173`, `storage.rs:160-176`). spamprobe's uncapped per-message frequency training (`FrequencyDB.cc:337-340`) is the weaker, higher-variance scheme. `cap=2` training would inflate the **spam** side of exactly the repeated ESP tokens → worse FP.
- **spamprobe does NOT tokenize a "rich header set by default."** The true no-flag default is `from/to/cc/subject/received` (`ParserConfig.cc:52-56`); the richer set (reply-to/sender/originator/message-id) is opt-in via `-H normal`. Return-Path and List-Id have **no spamprobe precedent by name**. Add #5 on its own merit, not "because spamprobe does it."
- **Be skeptical of tokenizing the full Received text** (a spamprobe behaviour). For THIS problem it likely makes FNs **worse** — it deposits ham weight on shared ESP-infrastructure tokens (relay banners, IPs, customer ids), the exact cross-contamination that is the root complaint. Experiment-only; if pursued, add the `IP_ADDRESS` collapse (`TraditionalMailMessageParser.cc:108-109`) to stop unique-IP blowup.
- **Don't "fix" a From-dilution that isn't happening.** The user's H1 mechanism is itself a misconception — spamlite keeps the whole address+domain prefix-isolated; spamprobe splits *more*. The fix is to add a generalizing anchor (#2), not to stop a non-existent overlap.
- **Don't expect the cold-start knobs (#7/#8/#9) to fix the supplier pin.** Those act on thin tokens; the discriminating ESP tokens are well-observed (n≥5). They cure H2, not the FP.
- **The tuner does NOT lack a holdout.** It already does a stratified 20% holdout and refuses to write params that don't generalize (`tune.rs` `stratified_split`, `main:515,533,537`). The genuine gap is narrower (full-corpus token-stat leak in `build_cache`); spec #11 against that, not against "add cross-validation."
