# 2026-04-30 — v0.4.0 cross-validation failure → v0.4.1 revert + per-user tuner

## What happened

The 1-week cluster shadow of v0.4.0 ended. Before deciding to roll v0.4.0
out as primary I cross-validated it against three real user databases on
mrn (`/tmp/eval/{wilira,georgivs,jaz}/db.sqlite`) using the
`spamlite-eval` harness against held-out portions of each user's own
corpus. Result: v0.4.0 was **worse than v0.3.0 on all three**.

The 54% error reduction reported on `admin_maildir` in the v0.4.0 work
was real but corpus-specific. `admin_maildir` has a thick `.TrainSpam/`
directory full of explicit ground-truth spam — most users don't have
that. The Phase 2 default flips (`strength 1.0 → 0.5`,
`unknown_prob 0.5 → 0.45`, `max_interesting 50 → 150`) made the
classifier more aggressive in a way that helped only when there was
enough training spam to support it. With the thinner per-user spam
counts that are typical, those settings just shifted borderline ham
into the spam half.

This is a textbook overfitting failure. The same-db eval methodology
caveat in the v0.3.0 journal turned out to be the entire story.

## Decision

Ship v0.4.1: revert the Phase 2 default flips, keep the v0.3.0
structural improvements (subject bigrams, `score`/`explain` split,
shadow mode wiring), and add an opt-in per-user tuning path so that
gains can be claimed where they actually generalise.

## What v0.4.1 contains

- **Reverted defaults** in `Params::default()`:
  - `strength` 0.5 → 1.0
  - `unknown_prob` 0.45 → 0.5
  - `max_interesting` 50 → 150

  These are the spamprobe-equivalent values. v0.4.1 is now algorithmically
  identical to v0.2.0 on a freshly-trained db; differences are structural
  (CLI shape, shadow mode, tokenizer edge-case fixes).

- **`Params::load_overrides(db_dir)`** in `src/classifier.rs`. Reads
  `<db_dir>/params.toml` at runtime — `key = value` pairs, `#` comments,
  whitelisted keys (`strength`, `unknown_prob`, `max_interesting`,
  `threshold`, `good_bias`, `min_word_count`). Missing file is silent;
  parse errors emit a `eprintln!` warning but never break delivery.
  Wired into `main.rs` via a `make_params()` helper that all four
  classification paths now use.

- **`spamlite-tune`** new binary (`src/bin/tune.rs`). Coordinate-descent
  parameter search over a real user db. Two non-negotiable design
  choices baked in to prevent another overfitting catastrophe:

  1. **Stratified split**, not mtime split. Train and test sets each
     preserve the global ham:spam ratio. Mtime split was tried first and
     produced laughable "93.8% improvement" numbers for jaz that
     immediately collapsed on a fresh holdout.
  2. **Balanced error rate** as objective:
     `(fp/ham_n + fn/spam_n) / 2`. Prevents the tuner from collapsing to
     "always predict ham" on imbalanced corpora (jaz has 31505 ham vs
     38 spam — naive total-error minimisation goes straight there).
  3. **Generalisation gate**: the tuner only writes `params.toml` when
     the tuned params beat the baseline on the held-out test set. If
     the test set doesn't improve, the tuner refuses to write anything
     and exits. This is the actual safety net.

- **UTF-8 fix** in `tokenizer::normalize_url`. URLs with multi-byte
  characters at the `max_len` boundary were panicking with "byte index N
  is not a char boundary; it is inside 'ф'". Fixed by walking back to
  the nearest `is_char_boundary`.

## Cross-validated tuner results

After fixing both the split and the objective, ran the tuner against
all three users:

| User      | baseline test bal-err | tuned test | verdict |
|-----------|-----------------------|-----------|---------|
| wilira    | 0.0016                | 0.0032    | refused — tuner found gains on train but they didn't hold |
| georgivs  | 0.0202                | 0.0158    | **wrote params.toml** — 21.6% real, generalising gain |
| jaz       | 0.1962                | 0.1962    | refused — params can't fix data hygiene |

The 19.6% baseline error on jaz is a **data problem**, not an
algorithm problem. He has flagged a substantial amount of normal mail
as spam over time, polluting the db; no amount of param tuning can
unmix the signal. Refusing to write fake gains is the right answer.

## Smoke test — params.toml is honored end-to-end

Staged `/tmp/spamlite-v0.4.1` on mrn pointing at `/tmp/eval/georgivs/`:

```
threshold = 1.001  →  verdict flipped from SPAM 1.000 to GOOD 1.000
```

Confirms `Params::load_overrides` is wired through every classification
code path.

## Deployed state

- `/usr/local/bin/spamlite-0.4.1` installed on mrn.
- `/usr/local/bin/spamfilter` patched: shadow candidate switched from
  `0.4.0` → `0.4.1`. `shadow.jsonl` entries now carry `"v":"0.4.1"`.
- `/etc/spamlite-shadow.allow` unchanged (still cluster-wide `*`).
- Primary path is still v0.2.0; v0.4.1 runs read-only as candidate.
- Verified live: a `receive` invocation shows
  `primary == cand == "GOOD 0.309680"` for a trivial test message — i.e.
  v0.4.1's output is bit-identical to v0.2.0 on default params, as
  intended.

## What this enables

The `cosmix-maild` rebuild can now embed v0.4.1 — it's drop-in
behaviourally compatible with v0.2.0 by default, and adds the
`params.toml` mechanism for the per-user tuning rollout.

## Lessons

- Same-db eval is a per-user *consistency* check, not a benchmark.
  Anything that promises a generalisation claim has to be cross-
  validated on held-out data with the original class balance preserved.
- "Improvement on corpus X" without a separate test corpus is a hint,
  not a result. v0.4.0 should have been blocked at the door for not
  having either holdout or per-user cross-validation.
- For imbalanced corpora, the only honest objective is balanced error.
  Total error optimisation will quietly maximise the majority class
  every time.
- A tuner that can refuse to write is more valuable than one that
  always finds something.
