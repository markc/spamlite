## Phase 0 — the training pipeline is disconnected

Original hypothesis from the plan: 215/216 users clustered near 5000/5000
is either a cap (inherited from Spamprobe's `maxHashTableSize`) or a
refreshable seed. Investigation rejected both.

- Spamprobe is **not** capped: `admin_maildir/.spamprobe.backup/sp_words` =
  880 KB (Jul 2025) vs `.spamprobe/sp_words` = 67 MB (Mar 2026). 77× growth
  under identical sieve wiring.
- Spamlite has no cap logic: read `src/storage.rs` top to bottom, `train()`
  is a straight `INSERT ON CONFLICT DO UPDATE`.
- Spamprobe's `-c` flag is **not** training mode: verified in
  `spamprobe.cc:625` it is just `setShouldCreateDbDir(true)`.
- Real sieve on mrn (`/etc/dovecot/sieve/global.sieve`) was `cat`'d and
  confirmed to call `spamfilter ... receive` for classification only.
  Training fires only on IMAP moves via `retrain-as-ham.sieve` and
  `retrain-as-spam.sieve`.

Root cause found in `spamprobe-1.4d/src/spamprobe/Command_receive.cc:104-121`:

> "receive — Scores message, prints score, and updates database accordingly."

Spamprobe's `receive` is unconditional TOE. `Command_receive::processMessage`
always calls `filter.classifyMessage(message, is_spam)` after scoring, which
writes token counts. **Spamlite's `receive` is implemented as spamprobe's
`score`** — pure classification, no DB update. That is the single biggest
functional difference between the two classifiers on this cluster. It
explains why `sp_words` grew 77× while `db.sqlite` stayed at 9 MB, why the
per-day stats JSONL shows corpus counts only moving on correction events,
and why the cluster is frozen at whatever the Spamprobe→spamlite migration
imported at baseline.

Cluster evidence from the 2026-04-14 digest confirmed the diagnosis:
multiple unrelated users show `spam=4929` exactly (cam@ck20, leathergoods,
walkenny) — fingerprint of a shared import baseline, not organic growth.
Good counts vary widely (2174, 3961, 4753, 5058, 29436) while spam clusters
tightly at 4900–5000 — asymmetric because users rescue fp from Junk (fires
`retrain-as-ham`, good grows) but rarely flag missed spam (retrain-as-spam
almost never fires). kingswaycomputers has 29436/4965 — decisive: even with
6× ham volume the spam side is still pinned at the plateau.

## v0.3.0 — confidence-gated training

Implemented spamprobe-style `receive` auto-training, but with a safety gate
instead of pure TOE. Pure TOE on a classifier with real fp/fn rates is a
self-reinforcing feedback loop: cam@ck20's 5 daily fp would get trained as
spam, producing more fp tomorrow. Confidence gate fixes this.

Changes in `src/main.rs`:

- Renamed old `cmd_receive` → `cmd_score`. The old behaviour (pure
  classification, no DB update) now lives under the correct name matching
  spamprobe's semantics. Kept for backward compat and read-only use.
- New `cmd_receive`: classify, print score to stdout first (preserves sieve
  contract), then train on self-verdict ONLY if score is outside the dead
  band `(gate, 1.0 - gate)`. Training failures are non-fatal stderr — mail
  delivery must not break because counts couldn't be written.
- New `-g GATE` flag + `SPAMLITE_TOE_GATE` env var, default `0.2`.
  Semantic: gate = trusted-region half-width. `gate=0.2` trains when
  `score ≤ 0.2 || score ≥ 0.8`. `gate=0.0` disables training. `gate=0.5`
  is pure TOE. Config-exposed from day one because the right gate width
  depends on the score distribution, which is an empirical question to be
  answered by the explain command once it exists.
- Renamed dispatcher entries: `receive` → new TOE variant, new `score`
  entry for the old behaviour. Usage text updated.
- Version 0.2.0 → 0.3.0.

Also earlier in the day (v0.2.0 portion, now folded in):

- `Verdict::Unsure` removed. Binary SPAM/GOOD.
- Collapsed `spam_threshold` (0.9) + `good_threshold` (0.1) into single
  `threshold` (default 0.5) with `-t` flag + `SPAMLITE_THRESHOLD` env.
- Dep bumps: `mail-parser` 0.10 → 0.11, `rusqlite` 0.32 → 0.38.

## Offline test loop against admin_maildir/

Validated end-to-end for the first time. Corpus sizes:
`Maildir/cur/` = 2531 messages, `.Sent/cur/` = 1548 (strong ham ground
truth), `.Junk/cur/` = 112 (known spam). `.spamlite/db.sqlite` starts at
5074/4947/255731 tokens per the live `.spamlite-stats.jsonl`.

Five-case gate test matrix against a sandbox copy of the db:

| Test | Setup | Expected | Result |
|---|---|---|---|
| A | default gate, 10 spam + 10 ham | +10/+10 | **+10/+10 ✓** |
| B | `-g 0`, empty DB, score=0.5 | blocked | **0/0 ✓** |
| C | default gate, empty DB, score=0.5 | blocked | **0/0 ✓** |
| D | `-g 0.49`, 10 spam + 10 ham | +10/+10 | **+10/+10 ✓** |
| E | `SPAMLITE_TOE_GATE=0.49`, 5 spam | +0/+5 | **+0/+5 ✓** |

Score stability: the admin_maildir messages all score at the extremes
(≤0.000006 for ham, ≥0.952 for spam). None land anywhere near the dead
band on the current corpus. This is good for baseline accuracy but does
mean TONE would be a near no-op on this cluster — confidence-gated TOE
is the right kickstart mechanism, not TONE.

Caught an inverted semantic in my own docstring during the test run:
I had documented `gate=0.0 = pure TOE` but the code (correctly) treats
`gate=0.0` as "nothing is trusted, no training" and `gate=0.5` as "trust
everything, pure TOE." Fixed before commit.

## Harvest v2 — delivery events + msgid pairing

The first harvest cut only captured TRAIN events, which don't include
message UIDs — so the event stream was a statistical summary of user
corrections, not a labelled corpus. Poking at the full log surface
revealed a much better data source: `global.sieve` emits a
`DEBUG: DELIVERY: user -> folder (VERDICT SCORE)` line for every
inbound message, followed within milliseconds by a sieve msgid=...
stored-into-mailbox line on the same session. Pair them by session id
and you get `(timestamp, user, verdict, score, folder, msgid)` for
every inbound message.

Rewrote `harvest.rs` to parse both event types. DELIVERY lines get
buffered in a per-session HashMap until the paired msgid arrives;
unpaired deliveries flush as orphans at end-of-input. 8 unit tests.

Ran against the full 31-day log history (`zcat -f /var/log/mail.log*`):

- **14,385 TRAIN events + 64,071 DELIVERY events** across 31 days
- 99.86% of deliveries paired with msgid (63,980 / 64,071)
- Zero parse failures across 2.1M log lines
- 192 unique delivery users out of 216 total on the cluster
- Average 2,067 events/day inbound

Notable per-user delivery spam rates:
- georgivs@nospam.com.au: 325 / 1592 (83% spam — absurd)
- admin@renta.net: 451 / 982 (69% spam — your own mailbox)
- admin@kingswaycomputers.com: 1367 / 1228 (47% spam)
- admin@wilira.com: 1176 / 922 (44% spam)

This is a small/medium business cluster with very high spam pressure
compared to general internet averages.

## Correlate binary — the labelled corpus

Added `src/bin/correlate.rs`, a second dep-free stdlib-only binary
that cross-references harvest events against a user's current Maildir
state to recover the confusion-matrix labels. Walks `Maildir/cur/`
and `Maildir/.X/cur/` (skipping `.Trash`), extracts the Message-Id
header from the first 16 KiB of each file, looks up the msgid in
the delivery event map, and emits one JSONL record per matched
message with the label:

- tp: delivery=SPAM and current=.Junk   (classifier correct)
- tn: delivery=GOOD and current=INBOX   (classifier correct)
- fp: delivery=SPAM and current=INBOX   (user rescued)
- fn: delivery=GOOD and current=.Junk   (user flagged)

Manual Message-Id header scanner handles folded continuation lines.
8 unit tests.

Live run against `admin_maildir/` with admin@renta.net's 1425 logged
deliveries produced **562 matched messages**:

| Label | Count | |
|---|---|---|
| tp | 95 | classifier correct on spam |
| tn | 436 | classifier correct on ham |
| fp | 16 | classifier wrong (user rescued) |
| fn | 15 | classifier wrong (user flagged) |

Raw numbers: **5.5% policy error rate** on admin's matched corpus.
Match rate was 562/1425 = 39%, which is the 31-day retention rate
— most unmatched files are older than the log window (the offline
admin_maildir has 4+ years of accumulated mail).

## Methodology fix — algorithm vs policy errors

The first correlate cut over-counted errors by conflating two
different things:

1. **Algorithm errors**: classifier was confident (score near 0 or 1)
   and the user disagreed. Fisher math produced a confidently wrong
   answer.
2. **Threshold-policy errors**: score was in the uncertain middle
   zone [0.2, 0.8] and the threshold in effect at delivery time
   determined the outcome. The threshold used to be 0.8 on this
   cluster (to accommodate users who never check their Junk folder),
   so a score of 0.75 was delivered as GOOD — and if the user later
   flagged it as spam, the naive correlator calls that "fn" even
   though the classifier wasn't confidently wrong.

Added `confidence` and `algo_error` fields to correlate output:

- confidence: "high_ham" (score ≤ 0.2), "uncertain" (0.2-0.8), or
  "high_spam" (score ≥ 0.8)
- algo_error: true iff classifier was confident AND user disagreed

Re-run on admin@renta.net:

- **11 algorithm errors** (2.0%), not 31 policy errors (5.5%)
- **ZERO algorithm-level fn** — the classifier never confidently
  misses spam on admin
- **All 11 algorithm-fp are commercial bulk mail from legit senders**:
  5x EasySuite digital signatures, 1x Klaviyo marketing, 1x Mailchimp,
  1x SendGrid, 1x Google bulk, 1x AWS EC2, 1x dovecot self-generated
  notification

The 2% algorithm error rate is the real classifier performance. The
other 3.5% is threshold-policy mismatch that per-user threshold
tuning (Phase 1.1) addresses directly. **Phase 2 algorithmic work
should target the 11 genuine algorithm errors, not the 31 policy
errors.**

## Methodology note — live-db explain is contaminated

Pointed `spamlite explain` at one of the 5 EasySuite fp messages
(delivery score 0.942, admin rescued). Current live db reports
**GOOD score=0.222** — a 0.72-point shift. That's not because the
algorithm fixed itself; it's because admin rescuing the message from
Junk fired `retrain-as-ham`, which trained this specific message's
tokens as ham into the live db. So explain shows the POST-correction
classifier state, not the state that produced the original error.

For Phase 2 falsification, use the **delivery-log score** as ground
truth, not `explain` against the live db. The log captures the
classifier's original output; the live db has been updated in-flight
by every subsequent user correction.

Side observation from that explain run: the top ham tokens in admin's
current db are dominated by Suncorp (Australian insurance/banking)
marketing email HTML/CSS fragments (`moz-hyphens`, `vml-p`,
`pp-icon-centre`, `si.suncorp.com.au`, etc). Admin received a lot of
Suncorp newsletters during the 2026-03-19 ham top-up pass, and their
boilerplate tokens became strong ham signals. New commercial bulk
mail from EasySuite/Klaviyo/Mailchimp/SendGrid shares similar HTML/CSS
structure but has sender-specific tokens with zero ham observations
— which produces the classic Bayesian failure mode: trained on one
bulk sender, still fires on different bulk senders. Phase 2.1
(`max_interesting: 150 → 27`) might mitigate this by weighting the
strongest sender-specific signals more heavily relative to diluted
boilerplate.
