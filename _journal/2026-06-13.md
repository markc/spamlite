## Found + fixed: threshold digest section dead since May 20

Today's digest email was 17,240 bytes vs a flat ~20.7–20.8 KB for weeks.
Bisection: `/usr/local/bin/spamlite-threshold-digest` exits 101 when run
with **no args** — mix 0.3.0 panics at `evaluator.rs:3812` ("is_stmt_sync
verified") evaluating `$BASE = $1` when `$1` is unbound. With an explicit
`/srv` arg it runs fine. Minimal repro: `$B = $1` in a no-arg script.

mail.log archaeology: digest sizes varied daily (21–23 KB = live threshold
content) through **May 19** — the exact mtime of `/usr/local/bin/mix` —
then went flat from May 20. So the mix binary swapped on May 19 introduced
the panic and **every digest since May 20 has been missing Section 1**
(threshold digest); the shadow section kept the email arriving, which
masked it. `set -o pipefail` + cron's pipe to `mail` meant the rc=101 was
swallowed.

**Fix (deployed):** `deploy/spamlite-daily-digest` now calls
`spamlite-threshold-digest /srv` explicitly; copied to
`mrn:/usr/local/bin/spamlite-daily-digest`, verified rc=0 with both
sections rendering (211 users, totals populated). Tomorrow's 04:30 email
is the real confirmation.

This is more ammunition for the planned mix upgrade on mrn at the next
shadow swap ([[mix_upgrade_mrn]]) — current mix exempts `$1` positionals
from unbound-variable errors, so the panic class doesn't exist there.

## Python→Mix port evaluation: spamlite-stats-collect (test case)

Full evaluation written up as
`_doc/2026-06-13-porting-python-to-mix-stats-collect.md` — a journaled
test case for "porting a production python3 script to Mix" generally.
Headlines:

- Every construct mapped and **probed against the 0.17.0 binary**: regex
  with capture groups + `(?im)` flags, `getopt`, `append_file`,
  compact `json_encode`, map-as-set inode dedupe, `fmt("%d")` truncation
  for score bucketing — all there.
- **Three Mix gaps found** (the test-case payoff, all → cosmix-lib-mix):
  no `stat()` (uid/gid/ctime/inode — the #1 Python→Mix friction point),
  no `chown()` (chmod exists, chown doesn't), and `read_file_bytes` has
  no max-bytes cap (header sniffing reads whole multi-MB messages).
- Workarounds verified live on mrn: `find -newercc STATSFILE -printf
  "%i\t%p"` reproduces the python ctime-cursor + inode dedupe exactly in
  one fork per folder; `stat -c '%u %g'` + `chown` + `timeout 30` cover
  the rest (~7 forks/user, same ballpark as the python's own forking).
- **Recommendation:** add `stat()`/`chown()`/read-cap to mix first (~1–2h,
  mrn redeploy is now trivial), port with `stat()` for metadata but keep
  `find` for the bulk folder scans, verify by field-wise diffing
  `--dry-run` output against the python's real JSONL for the same day,
  then swap the cron target keeping the python as `*.bak-py` for a cycle.
- Port size estimate: ~150–180 lines of Mix; runtime should stay ~90s.
