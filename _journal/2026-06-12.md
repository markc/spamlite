## Security audit of v0.4.1 (full codebase)

Findings, all fixed in v0.5.0 (`df65875`):

1. **`panic = "abort"` in the release profile** — this is what turned the
   tokenizer panic into SIGABRT/exit-134. Removed (unwind default restored).
2. **No panic safety net on the sieve hot path.** `receive`/`score` now wrap
   tokenize/classify in `catch_unwind` and fail OPEN with `GOOD 0.500000` +
   stderr log. Training is separately isolated after the verdict prints, so
   fail-open can never double-print.
3. **`Params::load_overrides` accepted out-of-range values** (negative
   strength, unknown_prob > 1 → nonsense math failing open silently). Now
   rejected with a warning, defaults kept.
4. **Meta counter lost-update race** — total_good/total_spam were
   read-modify-write; now a single atomic UPDATE (dovecot lmtp parallelises).
5. stdin capped at 64 MiB; CSV export skips tokens with embedded quotes or
   control chars; stale "0.4.0" usage string replaced with
   `env!("CARGO_PKG_VERSION")`; `-V/--version` added (we had been identifying
   deployed binaries by file size).

No SQL injection (fully parameterised), no other panic paths found in the
mail-path modules. Offline bins' unwraps are tests/arg-parsing only.
Deliberate non-fix: subject bigrams exceed the 40-char bound, but bounding
them now would orphan already-trained bigram tokens.

## v0.5.0 enhancement (one algorithmic change)

`u:` URL tokens are now extracted from **plain-text bodies** (`591d552`) —
previously only HTML bodies produced URL tokens, so text-only spam lost its
strongest signal. The u: namespace is warm in production dbs, so shadow can
judge it. Rejected flipping `expanded_headers` defaults: a score-only shadow
can't see gains from never-trained token namespaces — unjudgeable.

Benchmark (Mix script, fresh-train 700h/700s + 300/300 holdout from
admin_maildir, threshold 0.6): **exact parity** — fp=6 fn=28 both builds.
The corpus is HTML-heavy; production shadow is the real test.

## Dovecot edit-mail panic (found during the shadow assessment, now remediated)

- dovecot 2.4.1 lmtp on mrn panics at `edit-mail.c:1903
  (merge_modified_headers): assertion failed: (!edstream->parent_buffer)`
  while SAVING certain large multipart messages after global.sieve's
  `addheader :last "X-Spam-Status"`. Child killed with signal 6 → message
  never stored → upstream retries.
- Architecture insight: LMTP comes from **proxy.renta.net** (alias `proxy`,
  203.25.132.8 — its PTR falsely says mail.spiderweb.com.au), so the stuck
  message queues on the proxy, NOT mrn. Two episodes: May 22–27 (~40
  panics/day, ended at queue lifetime — that message likely bounced = silent
  mail loss) and June 9–12.
- June casualty: 428 KB ticketing email from smarttickets@airtickets.com.au
  to catherine@tewantintravel.com (a travel agent — real business mail),
  stuck 3 days. Hand-delivered via `doveadm save` (bypasses sieve, no panic),
  verified UID 40113 in INBOX, then `postsuper -d B3DD5631` on proxy.
- Repro sample kept at `mrn:/root/B3DD5631.eml` — no upstream report of this
  exact assertion exists; worth filing against dovecot/core with that sample.
  Same family as historic pigeonhole `addheader :last` stream bugs.
